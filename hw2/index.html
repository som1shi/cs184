Part 1: Bezier Curves with 1D de Casteljau Subdivision

De Casteljau's algorithm evaluates a point on a Bezier curve at some parameter t (between 0 and 1) through repeated linear interpolation. Given n control points, the algorithm repeatedly linearly interpolates between each adjacent pair at parameter t, reducing n points to n-1 each step until a single point remains — the location on the curve at t. We implemented this by defining a vector that stores the interpolated points for each step. For each adjacent pair of points, we compute (1 - t) * points[i] + t * points[i+1] and append the result. This is called repeatedly until one point remains.

Part 2: Bezier Surfaces with 2D de Casteljau Subdivision

De Casteljau's algorithm extends to Bezier surfaces by applying the same 1D algorithm twice: once in each direction of a 2D grid of control points. In this n×n grid, each row represents a Bezier curve in the u direction and each column represents a Bezier curve in the v direction. We implemented this using three functions: evaluateStep extends the 2D linear interpolation step to 3D by computing (1 - t) * points[i] + t * points[i+1] for each adjacent pair, evaluate1D repeatedly calls evaluateStep until one point remains, and evaluate applies evaluate1D across each row at u, collecting one point per row into a column, then calls evaluate1D on that column at v to produce the final surface point.

Part 3: Area-Weighted Vertex Normals

To compute the area-weighted vertex normal, we loop through all neighboring triangles of a vertex using the half-edge data structure. For each triangle, we take its three vertex positions and compute the cross product of two of its edges. This gives us a vector pointing perpendicular to the triangle whose length naturally scales with the triangle's area — so bigger triangles automatically contribute more to the final result. We sum these vectors across all neighboring triangles and normalize at the end to get the final unit normal. Part 4: Edge Flip

Implementation

The edge flip operation modifies the connectivity of a pair of triangles sharing an edge by reassigning the shared edge to connect the opposing vertices, without adding or removing any underlying mesh elements. We implemented this by first performing a boundary check to ensure the given edge and its adjacent faces are strictly interior. Next, we collected all elements in the local neighborhood—10 half-edges, 5 edges, 4 vertices, and 2 faces—using next() and twin() traversals. We then updated the mesh topology by calling setNeighbors(...) on the 6 inner half-edges to redefine the winding order of the two newly formed triangles. Finally, we explicitly reassigned the halfedge() pointers for every affected vertex, edge, and face to guarantee no dangling pointers were left referencing outdated topological states.

Debugging Journey: The "Black Hole" Incident

We had a bug where flipping an edge caused the triangles to disappear completely, leaving a "black hole" in the teapot. We traced this back to a fundamental misunderstanding of how vertices map to half-edges. We had written: 

VertexIter c = cb->vertex();
VertexIter a = ca->vertex();

Since a half-edge's vertex is its source, ca->vertex() evaluates to vertex c, not a. We was assigning the variables a and c to the exact same point in space! This caused my setNeighbors function to build degenerate, zero-area triangles that collapsed in on themselves, causing the renderer to cull them out as holes. Changing it to ab->vertex() fixed the winding order and instantly solved the problem.

Part 5: Edge Split

Implementation

The edge split operation inserts a new vertex at the midpoint of a shared edge and connects it to the opposing vertices, dividing the original two adjacent triangles into four. We implemented this by first calculating the new vertex position as the average of the original edge's endpoints, and then allocating the necessary new mesh elements: 1 vertex, 3 edges, 2 faces, and 6 half-edges. We exhaustively reassigned the pointers for all surrounding and newly created elements using setNeighbors(...) to properly wind counter-clockwise around the four new triangles.

Part 6: Subdivision for Mesh Upsampling (with Butterfly Extra Credit)

Implementation

Loop subdivision upsamples a mesh by subdividing each triangle into four smaller triangles and updating vertex positions based on a specific weighting scheme. We implemented this as a multi-step process to ensure the original mesh topology remained available for all weight calculations. First, we computed the updated positions for all existing vertices using a weighted average of their neighbors—specifically $(1 - n \cdot u) \cdot \text{original_position} + u \cdot \text{original_neighbor_sum}$, where $u$ is a constant based on the vertex degree $n$. Second, we calculated the positions for new vertices that would be inserted at edge midpoints using the formula $3/8 \cdot (A + B) + 1/8 \cdot (C + D)$ for the shared edge $(A, B)$ and opposite vertices $C$ and $D$. Once positions were cached, we performed 4-1 subdivision by splitting every original edge and flipping any new edge that connected an original vertex to a new vertex. Finally, we updated all vertex positions to their newly computed values.

  1. Calculate Positions: Calculate new positions for old vertices and new edge midpoints before modifying the mesh topology.
  2. Split: Iterate through a pre-saved list of original edges and split them.
  3. Flip: Iterate through all edges and flip any isNew edge that connects an old vertex to a new vertex.
  4. Finalize: Copy newPosition to position for all vertices.

For the extra credit, we extended our upsampler to support meshes with boundaries and implemented the Modified Butterfly subdivision scheme. To handle boundaries, we modified the vertex update rules: for a boundary vertex, the new position is computed using a 1D smoothing mask $3/4 \cdot v + 1/8 \cdot (b_1 + b_2)$, and new edge vertices are placed exactly at the midpoint $1/2 \cdot (A + B)$. Additionally, we implemented Butterfly subdivision, which is an interpolating scheme where original vertices remain at their initial positions. For new vertices, we implemented an 8-point stencil that reaches two rings out into the mesh, applying weights of $1/2$ to endpoints, $1/8$ to adjacent opposite vertices, and $-1/16$ to the four vertices in the secondary ring. This implementation successfully preserves the original vertex locations while generating a smooth, interpolated surface.

Debugging Journey: Loop Ordering

We encountered a severe bug where our upsampled mesh looked like a tangled ball of spikes. We realized we had accidentally placed the 8-point stencil calculation after the splitEdge loop. Because splitEdge alters the fundamental topology of the mesh, our h->next()->twin()->next() traversals were getting trapped inside the newly created subdivided triangles instead of reaching out to the next structural ring. Reordering the math to happen before the splits fixed the traversal instantly.

Observations on Mesh Behavior

Sharp Corners and Edges: When you subdivide a mesh, sharp corners and hard edges quickly lose their definition and smooth out into organic, rounded curves.

Reducing the effect: We can preserve sharp corners by heavily pre-splitting the edges immediately surrounding the corner before applying subdivision. Because Butterfly (and Loop) relies on the localized positions of neighboring vertices, drastically increasing the vertex density near a corner limits how far the subdivision stencil can "pull" the geometry away from its original sharp state.

The Asymmetric Cube

When you load dae/cube.dae and perform several iterations of subdivision, the cube becomes noticeably lopsided and asymmetric.

Why it occurs: A standard cube mesh is constructed using two triangles per face, meaning every square face is bisected by a single diagonal edge. This geometry is topologically asymmetrical. The subdivision stencil calculates positions based on edge connections; because the diagonal edge pulls the topological "weight" in one specific direction, the resulting subdivided surface bulges asymmetrically along those diagonals.

The Fix: To alleviate this, we can pre-process the cube by flipping/splitting the diagonal edge on every single face so that each square face contains four triangles meeting at a center vertex (forming an "X"). Once the base topology is perfectly symmetric across all axes, the subdivision algorithm applies its weights symmetrically, resulting in a perfectly smooth, evenly rounded cube!
